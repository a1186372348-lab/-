# 问题排查模板

出 bug 时不要直接让 AI 重写，先填这份模板，把问题变成一个可定位的工程问题。

---

## 1. 现象

只描述事实，不下结论。

模板：

- 用户做了什么：
- 实际发生了什么：
- 预期应该是什么：

## 2. 复现步骤

尽量写成任何人都能复现的步骤。

模板：

1. 打开哪里
2. 输入什么 / 点击什么
3. 等待多久
4. 看到什么错误

## 3. 问题类型

先分类，再排查。

- 编译错误
- 运行时报错
- 数据没写对
- UI 没更新
- 状态错乱
- 时序问题
- 外部 API 失败
- 多窗口联动错误

## 4. 影响范围

模板：

- 只影响当前页面 / 影响多个页面：
- 只影响展示 / 影响数据：
- 是否阻断主流程：

## 5. 证据

模板：

- 报错信息：
- 控制台日志：
- 网络请求：
- 数据库状态：
- 最近相关改动：

## 6. 先缩小范围

从下面 5 个方向缩小，不要一上来全盘重写。

### 输入

- 输入值是否正确？
- 参数是否为空或格式错误？

### 判断

- 条件分支是否进入了预期路径？
- 默认值是否干扰了结果？

### 状态

- 状态真相在哪？
- 是否出现了重复状态、旧状态、轮询覆盖？

### 输出

- UI 是否监听到了正确事件？
- 数据库或外部调用是否真的执行成功？

### 时序

- 是否存在延迟、轮询、流式回调、多窗口同步导致的竞态？

## 7. 最可能的模块

必须先给出 1-2 个候选，不允许“整个项目都有问题”。

模板：

- 首要怀疑：
- 次要怀疑：
- 怀疑理由：

## 8. 临时止血方案

如果还没修好，先写清楚怎么降低影响。

模板：

- 是否可回退：
- 是否可禁用入口：
- 是否可加保护判断：

## 9. 修复后必须补的验证

- [ ] 原始问题已复现并已消失
- [ ] 主路径未受影响
- [ ] 相邻功能未回归
- [ ] 同类边界场景再检查一次

---

## 你以后排查时的思维顺序

先确认问题真的存在，再确认它属于哪一层，最后才决定改哪段代码。

顺序固定为：

1. 复现
2. 分类
3. 找证据
4. 缩范围
5. 修复
6. 回归验证
