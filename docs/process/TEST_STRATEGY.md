# 测试策略

本文档定义 zhushou 项目的验证边界：哪些可以自动化检查、哪些需要人工验收、哪些当前不覆盖。

---

## 验证分层

| 层级 | 检查方式 | 触发时机 | 覆盖范围 |
|------|----------|----------|----------|
| **L1: 自动化静态检查** | TypeScript 编译 + Rust 编译 | 每次提交前 | 类型安全、编译错误 |
| **L2: 人工验收检查** | 启动 `npm run tauri dev` 肉眼确认 | 功能变更后 | 桌面交互、视觉、多窗口 |
| **L3: 故意不覆盖** | 无 | — | 需要特殊硬件/环境的场景 |

---

## L1: 自动化检查

自动化检查是每次改动的**最低门槛**，必须通过才能提交。

### 检查命令

```bash
# TypeScript 类型检查（前端变更必跑）
npm run check:ts

# Rust 编译检查（Rust/Tauri 变更必跑）
npm run check:rust

# 全量自动检查（推荐）
npm run check

# 验证入口（执行全量自动检查，通过后提示进入人工回归）
npm run verify
```

### `verify` 职责

`npm run verify` 是提交前的统一验证入口，职责为：

1. 执行全量自动化检查（`check:ts` + `check:rust`）
2. 自动检查通过后，开发者应继续按 [最小回归清单](./REGRESSION_CHECKLIST.md) 执行人工验收

`verify` 只覆盖编译期自动化检查，不替代人工回归。通过 `verify` 不等于功能验证完成。

### 覆盖范围

| 检查项 | 命令 | 能发现 | 不能发现 |
|--------|------|--------|----------|
| TypeScript 类型 | `npm run check:ts` | 类型错误、缺失导入、接口不匹配 | 运行时逻辑错误、异步时序问题 |
| Rust 编译 | `npm run check:rust` | 编译错误、类型不匹配、未定义引用 | 运行时 panic、系统 API 调用失败 |
| 前端构建 | `npm run build` | Vite 构建失败、资源缺失 | 运行时渲染错误 |

### 当前限制

- 项目当前没有单元测试、集成测试或 E2E 测试。
- `package.json` 中没有 `test` 脚本。
- 自动化检查只能捕获编译期错误，无法验证运行时行为。

---

## L2: 人工验收检查

以下场景**必须通过人工启动应用确认**，自动化检查无法替代。

### 必须人工验证的场景

| 场景 | 验证方式 | 涉及模块 |
|------|----------|----------|
| **多窗口行为** | 验证主窗口与各子窗口的创建、显示、隐藏、关闭是否正常 | App.tsx、各 Page 组件 |
| **透明度** | 验证窗口透明/半透明渲染是否正确，无闪烁或黑底 | CloudRenderer、App.tsx |
| **鼠标穿透** | 验证穿透模式切换：穿透时鼠标穿过云朵，非穿透时可交互 | App.tsx → Rust `set_window_passthrough` |
| **系统托盘** | 验证托盘图标显示、右键菜单、最小化到托盘 | Tauri 配置、lib.rs |
| **窗口位置联动** | 验证主窗口移动时子窗口跟随定位是否正确 | App.tsx 位置同步逻辑 |
| **动画效果** | 验证表情切换、行为动画（眨眼/张望/伸懒腰）、Framer Motion 过渡 | CloudRenderer、behaviorScheduler |
| **流式对话** | 验证 AI 对话流式输出、气泡显示/追加/关闭完整生命周期 | SpeechBubblePage、ai.ts |
| **声音播放** | 验证提醒音效和交互音效播放 | Howler 集成 |
| **专注计时** | 验证番茄钟计时、阶段切换、暂停/继续 | FocusPage |

### 人工验收最小步骤

1. 运行 `npm run tauri dev` 启动应用
2. 观察主窗口云朵是否正常渲染和动画
3. 悬停云朵，验证菜单是否出现，四个按钮是否可点击
4. 依次打开待办、设置、专注、定时任务子窗口，确认正常显示和关闭
5. 如果改动涉及特定功能，按对应需求卡的验收清单逐项确认

### 何时需要人工验收

- 修改了 `src/App.tsx` 中的窗口管理、事件监听、服务编排逻辑
- 修改了 `src/components/` 中任何组件的渲染或交互逻辑
- 修改了 `src-tauri/` 中的 Tauri command、窗口配置或托盘配置
- 修改了 CSS/动画相关的样式文件
- 修改了涉及 Tauri 事件通信的代码

## L3: 故意不覆盖

以下场景当前**不纳入常规验证**，原因是不影响核心功能或需要特殊环境。

| 场景 | 原因 |
|------|------|
| 高 DPI / 多显示器 | 开发环境为单显示器，无法覆盖多屏位置联动 |
| 低电量/性能受限 | 桌面应用在资源受限场景的表现需要专门测试环境 |
| 旧版 WebView2 | 仅测试当前安装的 WebView2 版本 |
| 网络异常（API 超时/断网） | AI 对话和天气服务依赖外部 API，网络异常处理不在当前优先级 |
| 并发写入冲突 | SQLite 单写模型下极少出现，当前不专门覆盖 |

---

## 检查执行规则

### 每次提交前

1. 根据改动范围运行对应的自动化检查（前端改动跑 `check:ts`，Rust 改动跑 `check:rust`，双侧改动跑 `check`）
2. 用 `git diff` 确认改动范围符合预期

### 功能变更后

1. 完成自动化检查
2. 启动 `npm run tauri dev` 进行人工验收
3. 按最小回归清单（`docs/process/REGRESSION_CHECKLIST.md`）逐项确认

### 跨模块改动

1. 运行 `npm run check`（全量自动化检查）
2. 人工验收覆盖所有受影响的模块
3. 重点关注事件通信链路：发送端 → 接收端是否完整

---

## 未来改进方向

以下是从"无自动化测试"向"有基本自动化测试"的演进路径，按优先级排序：

| 优先级 | 改进项 | 收益 |
|--------|--------|------|
| P1 | 为纯数据逻辑（记忆提取、去重、混合检索评分）添加单元测试 | 验证 AI 层核心算法正确性 |
| P2 | 为 Tauri command 添加集成测试 | 验证 Rust 侧系统能力的正确性 |
| P3 | 为关键 UI 组件添加渲染测试 | 验证组件渲染输出和交互行为 |
| P4 | 搭建 CI 自动运行 `npm run check` | 每次推送自动验证编译通过 |

当前阶段以人工验收为主，自动化改进在触及相关模块时逐步推进。
