# 技术债文档

> 本文档记录 zhushou 项目已确认的工程风险和待治理问题。
> 每条技术债标注严重程度，帮助区分"应尽快处理"和"可延后处理"。

---

## 严重程度定义

| 等级 | 含义 |
|------|------|
| **高** | 阻塞后续开发或持续恶化，应在下个迭代周期内处理 |
| **中** | 当前可工作但不健康，在触及相关模块时应顺手治理 |
| **低** | 已知问题但当前影响可控，可在合适时机统一处理 |

---

## TD-001: src/App.tsx 过重

| 字段 | 内容 |
|------|------|
| **严重程度** | 高 |
| **问题现象** | `src/App.tsx` 约 900 行，承担了过多职责：多窗口位置联动、打扰模式检测、光标轮询、事件监听注册、服务编排启动、AI 对话入口、气泡生命周期管理、提醒回调处理、定时任务回调处理等 |
| **当前影响** | 新增功能时容易继续堆积到此文件；修改一处有引入其他功能回归的风险；代码审查和定位问题成本高 |
| **建议方向** | 1. 将多窗口位置联动抽取为 `useWindowSync` hook；2. 将服务编排和启动逻辑抽取为 `useServices` hook；3. 将事件监听按域分组抽取；4. 将 AI 对话流程下沉至 `services/ai.ts` 或新模块 |
| **是否阻塞后续开发** | 是 — 每个新增功能都有"往 App.tsx 加代码"的惯性，不治理则持续恶化 |

---

## TD-002: 轮询偏多

| 字段 | 内容 |
|------|------|
| **严重程度** | 中 |
| **问题现象** | 项目中存在多个高频轮询：Todo 列表 1 秒轮询、屏幕颜色采样 1 秒轮询、Scheduler 60 秒轮询、屏幕监控 30 秒轮询、行为动画 10-30 秒随机轮询、光标位置轮询 |
| **当前影响** | 桌面应用常驻运行，多个 1 秒级轮询叠加对 CPU 和电池有持续消耗；Todo 的 1 秒轮询在无外部写入时完全浪费 |
| **建议方向** | 1. Todo 轮询改为事件驱动：外部写入后通过 Tauri 事件通知刷新；2. 颜色采样和光标轮询在窗口不可见时暂停；3. 评估哪些轮询可改为更长间隔或按需触发 |
| **是否阻塞后续开发** | 否 — 但在资源受限场景（低电量、多任务）下会显现性能问题 |

---

## TD-003: 事件边界松散

| 字段 | 内容 |
|------|------|
| **严重程度** | 中 |
| **问题现象** | 1. FocusPage 对 `focus-start/pause/reset/tick` 同时使用 `emit()` 和 `emitTo('main', ...)` 双重发送，语义重复；2. `todo-mouse-enter/leave` 只有发送端无监听端；3. `speech:hide` 和 `scheduler:reload` 只有监听端无发送端；4. speech 系列事件在 App.tsx 中使用 `emit()` 而非 `emitTo('speech-bubble', ...)` 广播；5. 早期事件未使用 `<domain>:<action>` 命名规范（`settings-changed`、`all-todos-complete`、`cc-event`） |
| **当前影响** | 孤立事件增加理解成本和维护困惑；`emit` 广播可能被非预期窗口接收；双重发送在功能上可工作但增加了调试难度 |
| **建议方向** | 1. FocusPage 去掉 `emit()` 重复发送，仅保留 `emitTo('main', ...)`；2. 确认 `todo-mouse-enter/leave` 是否有计划中的用途，无则清除；3. 补齐 `speech:hide` 和 `scheduler:reload` 的发送端；4. speech 事件改为 `emitTo('speech-bubble', ...)` 定向发送 |
| **是否阻塞后续开发** | 否 — 但新增事件时容易复制已有的不规范模式，继续扩大技术债 |

---

## TD-004: 状态真相可能重复

| 字段 | 内容 |
|------|------|
| **严重程度** | 中 |
| **问题现象** | 1. TodoPage 通过 1 秒轮询从 SQLite 拉取列表到 React state，存在内存中与数据库两份真相；2. Zustand store 存储 UI 状态（表情、天气、AI 处理中），但部分状态可能与组件本地 state 有交叉；3. App.tsx 中存在模块级变量存储高频轮询标志，与 React state 管理不一致 |
| **当前影响** | 数据库与内存状态可能短暂不一致（如轮询间隔内外部写入的待办不会立即显示）；Zustand 与组件 state 的边界不清晰，新开发者不确定状态该放在哪里 |
| **建议方向** | 1. 明确 Zustand 仅管 UI 状态，持久化数据以数据库为准（已部分落实在 ARCHITECTURE.md）；2. TodoPage 轮询改为事件驱动减少不一致窗口；3. App.tsx 模块级变量评估是否可迁移到 useRef 或 Zustand |
| **是否阻塞后续开发** | 否 — 但如果状态管理边界不固化，后续新增功能时状态放置会持续模糊 |

---

## TD-005: 人工验证成本高

| 字段 | 内容 |
|------|------|
| **严重程度** | 中 |
| **问题现象** | 项目当前无自动化测试。多窗口行为、透明度、鼠标穿透、系统托盘、窗口位置联动、动画效果等核心桌面交互全部依赖人工启动 `npm run tauri dev` 肉眼确认 |
| **当前影响** | 每次改动后的验证成本高，回归风险大；单人迭代时容易跳过验证步骤导致线上问题 |
| **建议方向** | 1. 建立最小回归清单（本 PRD US-014）；2. 对纯数据逻辑（记忆提取、去重、混合检索评分）考虑单元测试；3. 对 Tauri command 端到端流程考虑集成测试；4. UI 层面探索组件级测试覆盖关键渲染路径 |
| **是否阻塞后续开发** | 否 — 但长期来看，缺乏自动化测试会显著降低迭代信心和速度 |

---

## TD-006: Reminder 调度间隔与冷却阈值耦合

| 字段 | 内容 |
|------|------|
| **严重程度** | 低 |
| **问题现象** | Reminder 服务的调度间隔（多久检查一次）和冷却阈值（距上次提醒多久才能再次提醒）共用同一个数值，无法独立配置 |
| **当前影响** | 用户无法实现"每 5 分钟检查一次但 30 分钟内不重复提醒同一待办"这样的需求 |
| **建议方向** | 将调度间隔和冷却阈值拆分为两个独立设置项 |
| **是否阻塞后续开发** | 否 |

---

## TD-007: 数据库操作全在 JS 侧

| 字段 | 内容 |
|------|------|
| **严重程度** | 低 |
| **问题现象** | 所有数据库操作由 `src/services/db.ts` 通过 `tauri-plugin-sql` 在 JS 侧完成。Rust 侧 `db.rs` 仅保留说明注释，不参与数据库管理 |
| **当前影响** | 无即时风险，但 tauri-plugin-sql 的 API 限制了一些高级查询能力；大量原始 SQL 字符串分散在 JS 代码中，缺少编译时类型检查 |
| **建议方向** | 当前架构可继续运行，但应在 JS 侧建立更好的类型约束（如 sql template tag 或 query builder）；如果未来有性能敏感的查询，可考虑迁移到 Rust 侧 |
| **是否阻塞后续开发** | 否 |

---

## TD-008: AI 对话上下文组装集中度不足

| 字段 | 内容 |
|------|------|
| **严重程度** | 低 |
| **问题现象** | AI 对话的上下文组装（对话历史、日摘要、记忆检索、向量嵌入）分散在 `ai.ts` 和 `App.tsx` 中，上下文准备、API 调用、后处理（记忆提取、意图识别、历史压缩）的边界不够清晰 |
| **当前影响** | 修改对话流程时需要同时理解多个文件中的逻辑链路；后台异步任务（记忆提取、历史压缩）的触发点和错误处理不够显式 |
| **建议方向** | 1. 将上下文组装逻辑集中到 `ai.ts` 中的明确函数；2. 将后台异步任务（extractMemoriesAsync、compressHistoryAsync、extractScheduleIntentAsync）统一为可追踪的后台任务模式 |
| **是否阻塞后续开发** | 否 — 但在扩展 AI 能力时（如新增工具调用、多轮意图追踪），分散的上下文管理会增加复杂度 |
