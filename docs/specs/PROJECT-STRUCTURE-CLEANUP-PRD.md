# PRD: Zhushou 项目结构清理

## 介绍 / 概述

本 PRD 的目标不是新增终端用户功能，而是把 `zhushou` 仓库整理成一个结构清晰、职责边界明确、文档入口统一、验证路径固定的可持续维护项目。

当前仓库已经具备可运行的桌面应用能力，但在工程层面仍有明显问题：

- 根目录存在临时资产、工具资产、正式项目资产混放的情况。
- 文档已有一定基础，但缺少统一入口和项目当前状态的单一事实来源。
- `src/App.tsx` 过重，分层边界虽已被意识到，但还没有被系统化固化到文档和流程中。
- 检查与回归路径依赖人工记忆，缺少统一命令入口与最小回归清单。
- 主应用与 `cloudpet-mcp/` 的关系虽然存在事实，但对新接手的人并不够直观。

本次工作聚焦“结构清理与工程化补强”，不改变产品方向，不以大规模重构代码为目标，而是先建立一个适合单人长期迭代和 AI 协作的工程基础。

## 目标

- 建立清晰的仓库结构，明确根目录中各目录与关键文件的职责。
- 建立统一的项目状态入口文档，减少依赖聊天记录恢复上下文。
- 为现有核心能力补齐最小需求卡，明确目标、边界、输入输出与验收方式。
- 建立最小但够用的架构文档，明确 UI、协调层、服务层、Tauri/Rust 层与 MCP 子项目的职责。
- 建立统一的检查与验证入口，让开发者知道改动后该运行哪些命令、做哪些人工回归。
- 固化“后续新增功能先补需求卡、跨模块改动更新文档、避免继续堆积到 `src/App.tsx`”的协作规则。

## User Stories

### US-001: 盘点并清理仓库顶层结构
**描述：** 作为项目维护者，我希望仓库顶层目录能够清楚区分正式源码、子项目、文档、脚本和临时资产，以便我打开仓库后可以快速判断每个目录的用途。

**Acceptance Criteria：**
- [ ] 根目录中的每个顶层目录和关键散落文件都能在文档中找到用途说明。
- [ ] 对 `__MACOSX/`、`.agents/`、`.cursor/`、`scripts/ralph/`、`test-bubble.ps1` 等现有内容逐项给出“保留 / 移动 / 删除 / 忽略”的处理结论。
- [ ] `cloudpet-mcp/` 在文档中被明确标记为独立子项目或主项目附属子系统，不能继续保持语义模糊。
- [ ] 结构清理后，主应用源码目录、Tauri 工程目录、文档目录、脚本目录的边界比当前更清晰。
- [ ] 结构调整完成后查看 `git diff`，没有误删主业务源码、构建配置或数据库相关文件。

### US-002: 建立统一的项目状态入口
**描述：** 作为项目维护者，我希望有一份统一的项目状态文档，以便我不看聊天记录也能知道项目当前在做什么、下一步做什么。

**Acceptance Criteria：**
- [ ] 新增 `docs/PROJECT_STATUS.md`。
- [ ] 文档至少包含：项目当前目标、当前进行中事项、近期已完成事项、下一个最小动作、主要风险或阻塞。
- [ ] 文档内容基于当前项目事实填写，不允许只写空泛原则。
- [ ] 新接手开发者在 5 分钟内可以仅通过该文档理解“项目现在处于什么阶段”和“接下来最应该做什么”。

### US-003: 为现有核心能力补齐最小需求卡
**描述：** 作为开发者，我希望每个核心能力都有最小需求卡，以便我修改功能时知道它解决什么问题、输入输出是什么、边界在哪里。

**Acceptance Criteria：**
- [ ] 新增 `docs/specs/` 目录，作为功能需求卡主目录。
- [ ] 至少为以下能力建立需求卡：主窗口云朵、气泡窗口、Todo、Scheduler、Focus、Settings、AI 对话、Memory、Reminder。
- [ ] 每张需求卡至少包含：目标、用户路径、输入、输出、业务规则、边界与异常、当前状态、验收清单。
- [ ] 需求卡描述的是“功能应该做什么”，而不是“代码现在怎么写”。
- [ ] 阅读需求卡的人可以区分“需求边界”和“实现细节”。

### US-004: 建立功能状态矩阵和路线图
**描述：** 作为项目维护者，我希望知道哪些能力已经上线、哪些正在进行、哪些只是想法，以便后续开发有明确优先级而不是发散推进。

**Acceptance Criteria：**
- [ ] 新增 `docs/specs/FEATURE_MATRIX.md`。
- [ ] 新增 `docs/specs/ROADMAP.md`。
- [ ] `FEATURE_MATRIX.md` 至少包含：功能名、当前状态、优先级、关联文档、备注。
- [ ] `ROADMAP.md` 至少包含：当前阶段重点、明确暂不做事项、下一阶段候选工作。
- [ ] 状态枚举统一使用固定集合，例如 `idea / planned / in-progress / shipped / paused`，避免同义词混用。

### US-005: 建立最小架构文档
**描述：** 作为开发者，我希望知道新代码应该放在哪一层、哪些能力由哪一层负责，以便不再继续把逻辑堆进单一文件。

**Acceptance Criteria：**
- [ ] 新增 `docs/architecture/ARCHITECTURE.md`。
- [ ] 新增 `docs/architecture/EVENTS.md`。
- [ ] 新增 `docs/architecture/DATA_FLOW.md`。
- [ ] `ARCHITECTURE.md` 明确说明 UI 层、页面协调层、服务层、Tauri/Rust 层、`cloudpet-mcp/` 的职责和边界。
- [ ] `EVENTS.md` 列出当前关键事件的发送方、接收方、用途，并与现有事件命名保持一致。
- [ ] `DATA_FLOW.md` 至少覆盖 Todo、Reminder、Scheduler、Chat、Memory 这几条关键链路。
- [ ] 文档中明确标出 `src/App.tsx` 是当前重点治理对象，并写清后续新增逻辑不应继续默认堆入该文件。

### US-006: 记录当前明确存在的技术债
**描述：** 作为项目维护者，我希望把已经确认存在的问题记录下来，以便后续治理时有清晰依据，而不是依赖印象。

**Acceptance Criteria：**
- [ ] 新增 `docs/architecture/TECH_DEBT.md`。
- [ ] 文档至少记录这些问题：`src/App.tsx` 过重、轮询偏多、事件边界松散、状态真相可能重复、人工验证成本高。
- [ ] 每条技术债至少包含：问题现象、当前影响、建议处理方向、是否阻塞后续开发。
- [ ] 读者能够区分“应该尽快处理的问题”和“可以延后处理的问题”。

### US-007: 建立最小测试策略与回归清单
**描述：** 作为开发者，我希望知道哪些改动需要自动检查、哪些改动必须人工验收，以便每次修改后都能做稳定回归。

**Acceptance Criteria：**
- [ ] 新增 `docs/process/TEST_STRATEGY.md`。
- [ ] 新增 `docs/process/REGRESSION_CHECKLIST.md`。
- [ ] `TEST_STRATEGY.md` 明确区分：自动化检查范围、人工验收范围、暂不覆盖范围。
- [ ] `REGRESSION_CHECKLIST.md` 至少覆盖：主窗口显示、子窗口打开、Todo 基本操作、Scheduler 刷新、Focus 状态切换、气泡显示。
- [ ] 文档明确写出多窗口、透明、穿透、托盘、定位联动属于人工重点验收项。

### US-008: 统一开发与检查命令入口
**描述：** 作为开发者，我希望根目录提供统一的命令入口，以便完成改动后不用靠记忆拼装检查命令。

**Acceptance Criteria：**
- [ ] 根目录 `package.json` 中新增或整理统一脚本入口。
- [ ] 至少存在 `check:ts`、`check:rust`、`check`、`verify` 这几类脚本。
- [ ] `check:ts` 可执行 `npx tsc --noEmit`。
- [ ] `check:rust` 可执行 `cargo check --manifest-path src-tauri/Cargo.toml`。
- [ ] `check` 至少能覆盖当前最小必要的自动化检查。
- [ ] `verify` 的职责在文档中被清晰定义，例如“执行自动检查并提示需要人工回归的场景”。

### US-009: 重组文档目录并修复入口导航
**描述：** 作为项目维护者，我希望文档目录按用途分层，以便任何人都能快速找到需求、架构、流程和日志，而不是靠猜文件名。

**Acceptance Criteria：**
- [ ] `docs/` 目录被整理为清晰分区，例如 `specs/`、`architecture/`、`process/`、`logs/`。
- [ ] 现有文档迁移或重组后，所有入口文档中的引用路径都被同步更新。
- [ ] `DEVELOPMENT_GUIDE.md` 被更新为新的文档导航入口。
- [ ] 文档之间不存在指向已不存在文件的死链接。

### US-010: 固化后续代码治理规则
**描述：** 作为开发者，我希望在本轮结构整理后有明确的持续治理规则，以便项目不会很快回到混乱状态。

**Acceptance Criteria：**
- [ ] 文档中明确规定：新增功能前先补需求卡，再进入实现。
- [ ] 文档中明确规定：跨模块改动后要同步更新相关文档。
- [ ] 文档中明确规定：新业务逻辑默认进入服务层或合适模块，不继续优先堆入 `src/App.tsx`。
- [ ] 文档中明确规定：提交前至少运行最小检查并查看 `git diff`。
- [ ] 这些规则出现在明确的开发入口文档中，而不是散落在多份说明中。

### US-011: 建立阶段性交付顺序
**描述：** 作为项目维护者，我希望这次结构清理有明确阶段顺序，以便执行时优先做“先稳定地基、再补文档、最后补流程”的工作，而不是并行发散。

**Acceptance Criteria：**
- [ ] PRD 中明确拆分至少 3 个阶段，例如“结构盘点与清理”“文档体系补齐”“验证与流程固化”。
- [ ] 每个阶段都写明产出物和完成标志。
- [ ] 后一阶段依赖前一阶段时，依赖关系在 PRD 中被显式写明。
- [ ] 执行者可以按阶段逐步交付，而不是要求一次性完成全部清理工作。

## Functional Requirements

- FR-1: 系统必须提供一份根目录结构说明，解释仓库顶层目录和关键文件的用途。
- FR-2: 系统必须提供一份统一的项目状态文档，作为当前项目事实入口。
- FR-3: 系统必须为现有核心能力建立最小需求卡并集中管理。
- FR-4: 系统必须提供功能状态矩阵，标识能力当前状态和优先级。
- FR-5: 系统必须提供路线图文档，说明当前阶段重点与后续候选工作。
- FR-6: 系统必须提供架构分层说明文档，明确模块职责边界。
- FR-7: 系统必须提供事件文档，记录关键事件的命名、方向和用途。
- FR-8: 系统必须提供关键数据流文档，帮助理解主要功能链路。
- FR-9: 系统必须提供技术债文档，记录当前已知工程风险。
- FR-10: 系统必须提供测试策略文档，区分自动检查与人工验收边界。
- FR-11: 系统必须提供最小回归清单，覆盖当前核心桌面交互链路。
- FR-12: 系统必须在根目录提供统一的开发检查命令入口。
- FR-13: 系统必须更新开发入口文档，使文档导航和执行顺序清晰可查。
- FR-14: 系统必须固化“新增功能先补需求卡、跨模块改动更新文档、避免继续堆积到 `src/App.tsx`”等治理规则。
- FR-15: 系统必须将本次结构清理拆分为可阶段性交付的执行计划。

## Non-Goals

- 不新增新的终端用户功能。
- 不大规模重写现有 React、Rust 或数据库实现。
- 不要求本轮彻底完成 `src/App.tsx` 的代码级拆分。
- 不要求本轮建立完整企业级 CI/CD 流程。
- 不要求本轮把所有模块补齐高覆盖率自动化测试。
- 不调整现有产品方向、交互策略或 AI 能力范围。

## Design Considerations

- 本次重点是信息结构设计，而不是视觉界面改版。
- 所有新增文档应优先服务“新人 5 分钟理解项目结构”和“AI agent 低歧义执行”两个目标。
- 文档目录分层应尽量稳定，避免后续频繁重命名造成引用漂移。

## Technical Considerations

- 仓库是 `Tauri 2 + React + TypeScript + Rust + SQLite` 的混合工程，结构清理不能破坏前后端联动关系。
- 多窗口、透明、穿透、托盘、位置联动等能力高度依赖 Tauri 运行时，不能只用静态文档推导，需要保留人工验收入口。
- `cloudpet-mcp/` 是独立 Node 子项目，整理时应避免与主桌面应用的改动混在一起。
- 根目录当前没有正式的 `test` 或 `lint` 统一入口，整理时应优先补“最小可执行检查链路”而不是一步到位搭建完整测试框架。
- 读取和维护中文文档时需统一使用 UTF-8，避免再次出现编码污染。

## Success Metrics

- 新接手开发者能在 5 分钟内理解仓库顶层结构、当前项目状态和下一步动作。
- 开发者在完成常规改动后，能明确知道先运行哪个检查命令、哪些场景需要人工回归。
- 核心能力都有可阅读的最小需求卡，不再需要从代码中反推功能边界。
- 架构分层、事件、关键数据流有统一文档入口，不再依赖分散知识点。
- 后续新增需求有明确入口规则，项目结构不再快速回到混乱状态。

## Open Questions

- `.agents/` 是否应被视为长期保留的项目协作资产，还是仅作为本地工作区工具目录处理？
- `.cursor/` 是否需要纳入正式仓库治理范围，还是仅作为本地编辑器配置忽略？
- `scripts/ralph/` 是否属于主项目长期工具链的一部分，还是应迁移到独立工具目录或子项目？
- `cloudpet-mcp/` 后续是持续与主应用同仓维护，还是计划拆出为独立仓库？
- `docs/SESSION_LOG.md` 等过程型文档是否需要迁移到 `docs/logs/`，以及迁移时机是否放在本次清理内？

## 分阶段执行建议

### Phase 1: 结构盘点与顶层清理
- 产出物：根目录结构说明、无效或临时资产处理结论、`docs/PROJECT_STATUS.md`
- 完成标志：根目录用途清晰，当前项目状态入口建立完成

### Phase 2: 文档体系补齐
- 产出物：`docs/specs/`、`docs/architecture/`、`docs/process/` 下的核心文档
- 完成标志：核心能力、架构边界、技术债、测试策略都有明确文档承载

### Phase 3: 检查入口与治理规则固化
- 产出物：统一检查脚本、更新后的 `DEVELOPMENT_GUIDE.md`、明确的协作规则
- 完成标志：改动后检查路径清晰，后续结构治理规则可被持续执行
