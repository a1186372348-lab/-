# 项目总览

## 项目定位

这是一个桌面宠物助手项目，技术栈是 `Tauri 2 + React + TypeScript + Rust + SQLite`。当前阶段已经不是“空壳原型”，而是多个功能已经接通的 MVP：

- 主窗口云朵角色
- 气泡子窗口
- 待办管理
- 定时提醒
- 专注计时
- 屏幕感知后的主动发言
- AI 对话、记忆、日摘要

## 你应该先理解的 4 层

### 1. UI 层

职责：负责显示、交互、动画，不负责复杂业务判断。

典型位置：

- `src/components/`
- `src/main.tsx`

关键认知：

- 页面只是“让用户操作”和“展示状态”。
- 页面里可以做轻量事件处理，但不应该塞入复杂规则。

### 2. 页面协调层

职责：把多个窗口、服务和事件串起来。

典型位置：

- `src/App.tsx`

关键认知：

- 这里是当前项目最重的文件。
- 它负责窗口位置、气泡显示、提醒联动、屏幕监控、设置变化监听。
- 它现在能工作，但说明项目已经出现“主入口过重”的维护风险。

### 3. 业务服务层

职责：封装 AI、提醒、调度、天气、数据库访问等业务能力。

典型位置：

- `src/services/ai.ts`
- `src/services/db.ts`
- `src/services/reminder.ts`
- `src/services/scheduler.ts`
- `src/services/screenMonitor.ts`

关键认知：

- 业务规则尽量放在这里，而不是组件里。
- 如果一个功能涉及“查询数据 + 判断规则 + 写入结果”，优先考虑放 service。

### 4. 系统能力层

职责：访问操作系统原生能力。

典型位置：

- `src-tauri/src/commands/system.rs`
- `src-tauri/tauri.conf.json`

关键认知：

- 这里处理鼠标位置、截图、全屏检测、窗口穿透。
- 凡是浏览器环境做不到、只能依赖系统 API 的事情，都应该往这层放。

## 当前项目的关键数据流

### 流程 1：用户聊天

1. 用户在输入框发送文本。
2. `App.tsx` 调用 `chatStream`。
3. `ai.ts` 读取设置、加载历史、查询记忆、流式请求模型。
4. 返回内容实时追加到气泡窗口。
5. 完成后写入 `chat_history`，并异步尝试提取记忆、定时意图。

你要学会问自己：

- 输入是什么？
- 中间加工在哪一层？
- 数据持久化在哪发生？
- UI 更新是主动拉取还是事件推送？

### 流程 2：待办与提醒

1. 待办页从 `db.ts` 读取 `todos`。
2. 用户增删改查都先调 DB service。
3. 提醒服务定时检查未完成任务。
4. 找到目标后更新 `last_reminded_at` 并通知主窗口。
5. 主窗口播放气泡、表情、音效。

你要学会问自己：

- 状态真相在哪？
- 页面只是展示，还是私自维护了一份重复状态？
- 这个功能改动会不会影响提醒逻辑？

### 流程 3：定时任务

1. 用户在定时页手动创建任务，或聊天时被 AI 识别出定时意图。
2. `scheduled_tasks` 落库。
3. `scheduler.ts` 轮询判断 daily / interval 是否触发。
4. 触发后更新 `last_triggered_at`，再通知主窗口。

你要学会问自己：

- 触发条件是谁判断的？
- 如何防止重复触发？
- 改了调度规则后，页面展示和数据库字段是否也要一起改？

## 当前项目暴露出的典型工程问题

### 1. 主入口过重

`src/App.tsx` 同时负责：

- 多窗口控制
- 气泡控制
- 提醒联动
- 屏幕监控接入
- focus 事件同步
- 设置变更监听

这说明项目能跑，但复杂度已经上来了。你以后看到“一个文件承担太多职责”，要先想到拆分，而不是继续往里加。

### 2. 验证链路弱

项目已接了不少能力，但当前没有成体系的：

- 单元测试
- 集成测试
- 固定回归清单
- 发布前 checklist

这会导致“功能看起来实现了，但改一下就回归”。

### 3. 轮询偏多

当前有一些“为了先跑通”而采用的轮询式方案，比如页面同步和调度检查。MVP 阶段可以接受，但你需要知道：

- 轮询常常是阶段性方案
- 轮询带来时序问题、重复触发风险、性能浪费
- 后续应逐步替换成事件驱动或更清晰的同步机制

### 4. 工作区治理不足

当前仓库曾出现构建产物、临时文件、本地脚本未被管理的情况。这属于典型新手问题，不是因为你差，而是因为你还没建立“交付卫生习惯”。

## 你以后读代码时必须回答的 4 个问题

1. 这个文件的唯一职责是什么？
2. 它依赖谁，谁又依赖它？
3. 它读写的状态真相在哪里？
4. 如果这里出 bug，我先看输入、判断还是输出？

只要你坚持这样读项目，你会比“只会问 AI 这段代码啥意思”进步快得多。
