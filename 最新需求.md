# 助手产品六层优化任务分解（自动化细化前置方案）

## 1. 文档目的

本文件用于把当前助手产品的整体优化拆成 6 个层面，并进一步细化为可独立推进的子任务，为后续自动化程序继续拆分 story / 子任务提供稳定输入。

当前约束：

- 本轮不动代码
- 本轮不改现有功能行为
- 本轮不做数据库调整
- 本轮不直接进入实现
- 只输出后续可执行的治理地图

使用方式：

- 每次只选择一个层面推进
- 每个层面先完成边界定义，再拆成 story
- 自动化程序后续应基于本文件继续向下细化，而不是重新发散讨论

---

## 2. 六层总览

### Layer 1：架构与代码层
目标：让项目从“主入口集中编排”升级为“多领域分层协作”。

### Layer 2：事件与状态层
目标：让跨窗口通信和状态归属从“经验规则”升级为“显式模型”。

### Layer 3：AI 交互层
目标：让 AI 能力从“能跑”升级为“可扩展、可追踪、可治理”。

### Layer 4：产品体验层
目标：让助手从“功能集合”升级为“连续存在、节奏自然的桌面陪伴产品”。

### Layer 5：工程与质量层
目标：让项目从“能编过 + 靠人工确认”升级为“关键行为有最小自动化保障”。

### Layer 6：商业与扩展层
目标：先定义未来扩展边界，为插件化 / 商业化 / MCP 能力演进预留空间。

---

## 3. Layer 1：架构与代码层

### 3.1 层级目标

把当前助手项目从“主入口知道一切、亲自协调一切”，优化成“多领域子系统协作，主入口只做薄协调”。

### 3.2 当前现状

- `src/App.tsx` 仍承担窗口编排、服务启动、事件监听、AI 表现联动、低干扰模式、气泡控制等多重职责
- 代码分层意识已经存在于文档中，但代码本体未完全落实
- 多窗口逻辑虽然可运行，但复用性、可维护性和扩展性较弱
- 新能力继续默认堆进 `App.tsx` 的风险仍然很高

### 3.3 子任务清单

#### 任务 1.1：主协调层职责盘点
- 目标：
  - 把 `App.tsx` 中现有职责完整分类，建立后续拆分边界图
- 任务内容：
  - 分类现有职责：
    - 窗口编排
    - 服务启动
    - 事件桥接
    - AI 表现
    - UI 本地状态
    - 低干扰 / 穿透控制
  - 标记每项职责未来应归属哪一层
- 完成标志：
  - 形成无歧义的职责总表
  - 每项职责只有一个主归属
- 关键改动内容（未来实现时）：
  - 不直接改逻辑，先建立拆分清单

#### 任务 1.2：薄协调层结构定义
- 目标：
  - 明确未来 `App.tsx` 只保留哪些职责
- 任务内容：
  - 定义薄协调层只保留：
    - 页面组装
    - 用户交互入口
    - 少量 UI state
    - 调用各领域 hook / controller
  - 明确禁止继续进入 `App.tsx` 的逻辑类型
- 完成标志：
  - 后续实现者不再需要自行判断“什么该留在 App.tsx”
- 关键改动内容（未来实现时）：
  - 将新增 runtime / event / window / AI 逻辑全部导向子层

#### 任务 1.3：窗口编排层设计
- 目标：
  - 把 todo/settings/focus/scheduler/speech-bubble 的控制抽离为独立层
- 任务内容：
  - 定义负责内容：
    - show/hide
    - 位置同步
    - bounds 更新
    - hover 延时
    - 子窗口互斥
    - 光标轮询
  - 定义输出接口为意图级动作，而不是底层 API
- 完成标志：
  - 后续实现时可直接拆出窗口编排 hook / coordinator
- 关键改动内容（未来实现时）：
  - 将窗口位置、可见性、互斥逻辑从 `App.tsx` 迁出

#### 任务 1.4：应用运行时层设计
- 目标：
  - 把 weather/reminder/scheduler/screenMonitor/timeCycle/colorSampler 的生命周期统一归位
- 任务内容：
  - 定义 runtime 层负责：
    - 初始化
    - 清理
    - 配置变更响应
    - 回调派发
  - 明确常驻服务与条件服务边界
- 完成标志：
  - 后续实现时可将启动/清理逻辑整体迁出 `App.tsx`
- 关键改动内容（未来实现时）：
  - 形成统一 runtime hook / controller

#### 任务 1.5：分层实施顺序设计
- 目标：
  - 明确未来重构时的分步顺序，避免大爆炸
- 任务内容：
  - 规定实施顺序：
    1. 窗口编排层
    2. runtime 层
    3. 事件桥接层
    4. AI 表现协调层
  - 明确每一步必须保持哪些行为不变
- 完成标志：
  - 自动化程序可直接按顺序拆 story
- 关键改动内容（未来实现时）：
  - 分阶段迁移，不做一次性重构

## 4. Layer 2：事件与状态层

### 4.1 层级目标

让事件模型、跨窗口通信和状态真相从“依赖经验”变成“依赖显式规则”。

### 4.2 当前现状

- `emit` / `emitTo` 仍存在历史混用
- 有孤立事件、重复语义事件、历史命名遗留
- React state、Zustand、SQLite、模块级变量的职责边界未完全硬化
- 部分系统状态依赖多个布尔变量拼装

### 4.3 子任务清单

#### 任务 2.1：事件总表清理方案
- 目标：
  - 形成“保留 / 收敛 / 淘汰”的事件决策表
- 任务内容：
  - 分类现有事件：
    - 保留
    - 重命名
    - 定向化
    - 移除
    - 补发送端 / 监听端
  - 标记广播事件与定向事件边界
- 完成标志：
  - 每个事件都有明确去向
- 关键改动内容（未来实现时）：
  - 收缩历史遗留事件集合

#### 任务 2.2：跨窗口通信模型设计
- 目标：
  - 统一主窗口、子窗口、speech bubble、Rust bridge 的通信规则
- 任务内容：
  - 规定：
    - 什么情况下用 `emitTo`
    - 什么情况下允许广播
    - 主窗口是否作为统一中枢
- 完成标志：
  - 新增事件无需再次选模式
- 关键改动内容（未来实现时）：
  - 重构事件方向图和调用点

#### 任务 2.3：状态真相分层方案
- 目标：
  - 明确 UI 状态、运行时状态、持久化状态的边界
- 任务内容：
  - 定义：
    - Zustand：UI / 展示状态
    - 组件 state/ref：局部短生命周期状态
    - SQLite：持久化业务真相
    - 模块级变量：逐步迁往更明确的容器
- 完成标志：
  - 后续任何状态都能直接判断归属
- 关键改动内容（未来实现时）：
  - 减少状态重复真相

#### 任务 2.4：系统状态模型显式化
- 目标：
  - 把零散布尔值状态收敛成明确状态域
- 任务内容：
  - 定义至少三类状态域：
    - 面板/窗口状态
    - disturbance / presence 状态
    - speech 状态
- 完成标志：
  - 状态流转不再依赖隐式组合
- 关键改动内容（未来实现时）：
  - 用统一模型替代多处布尔拼接

#### 任务 2.5：事件与状态治理规则固化
- 目标：
  - 给后续新增事件和状态建立固定规则
- 任务内容：
  - 定义新增事件 checklist
  - 定义新增状态 checklist
  - 定义禁止延续旧模式的条款
- 完成标志：
  - 自动化程序后续可直接拿规则做校验
- 关键改动内容（未来实现时）：
  - 更新事件/状态设计规范

## 5. Layer 3：AI 交互层

### 5.1 层级目标

把 AI 相关能力从“聊天能跑 + 主入口联动展示”升级为“输入、决策、展示、后台任务分离”的可扩展助手架构。

### 5.2 当前现状

- 聊天流、气泡流、表情流、记忆提取、意图识别之间耦合较紧
- AI 的产出与 UI 表现没有完全解耦
- 后台任务边界、失败处理、重试策略没有形成稳定模型
- 未来新增工具调用、多轮意图、角色系统时复杂度会快速上升

### 5.3 子任务清单

#### 任务 3.1：AI 交互链路盘点
- 目标：
  - 画清楚从用户输入到气泡输出的完整链路
- 任务内容：
  - 盘点：
    - 输入进入点
    - stream 输出
    - speech append/done
    - expression 切换
    - memory extraction
    - schedule intent extraction
    - history compression
  - 区分同步链路和异步后台链路
- 完成标志：
  - 能清楚回答每条 AI 能力链路从哪来、到哪去
- 关键改动内容（未来实现时）：
  - 为后续解耦打基础

#### 任务 3.2：消息流与表现流解耦方案
- 目标：
  - 让 AI 负责“生成什么”，UI 负责“怎么展示”
- 任务内容：
  - 定义：
    - message flow：文本 / 意图 / 命令
    - presentation flow：气泡 / 表情 / 时长 / 动画
  - 明确谁负责转换
- 完成标志：
  - 后续新增 AI 能力无需直接改 UI 展示链
- 关键改动内容（未来实现时）：
  - 建立 AI 输出到展示层的稳定接口

#### 任务 3.3：后台任务模型设计
- 目标：
  - 把记忆提取、摘要压缩、意图识别统一成后台任务模型
- 任务内容：
  - 定义后台任务的：
    - 触发条件
    - 输入
    - 输出
    - 失败处理
    - 重试策略
    - 是否阻塞主对话
- 完成标志：
  - 后续实现时后台任务不再散落在主流程中
- 关键改动内容（未来实现时）：
  - 后台任务统一归位

#### 任务 3.4：助手主动性策略层设计
- 目标：
  - 把主动发言从“多个触发点集合”升级成“统一决策策略”
- 任务内容：
  - 分类主动发言来源：
    - screen monitor
    - reminder
    - scheduler
    - cc-event
    - timeCycle
  - 定义优先级、冲突规则、打断规则
- 完成标志：
  - 后续主动发言不会互相抢占和混乱叠加
- 关键改动内容（未来实现时）：
  - 统一主动提醒调度逻辑

#### 任务 3.5：AI 能力扩展边界设计
- 目标：
  - 为未来工具调用、多轮意图、角色系统预留结构空间
- 任务内容：
  - 定义：
    - 核心对话能力
    - 扩展 AI 能力
    - 哪些能力不允许直接进入主流程
- 完成标志：
  - 后续扩展不会继续把主流程做成大杂烩
- 关键改动内容（未来实现时）：
  - AI 扩展通过边界接入，而不是直接塞主入口

## 6. Layer 4：产品体验层

### 6.1 层级目标

把助手从“功能集合”优化成“连续存在、节奏自然、不过度打扰的桌面陪伴产品”。

### 6.2 当前现状

- 功能已有基础，但整体体验更偏功能点驱动
- 提醒、主动发言、气泡、表情、专注、待办之间缺统一体验节奏
- 人设、语气、打扰程度、提示密度尚未完全形成产品级一致性

### 6.3 子任务清单

#### 任务 4.1：助手角色与语气统一方案
- 目标：
  - 定义助手的人设、语气范围、情绪边界
- 任务内容：
  - 明确：
    - 助手更偏陪伴、执行还是提醒
    - 什么时候温柔
    - 什么时候直接
    - 什么时候不说话
- 完成标志：
  - 后续所有文案和提示都能落在统一角色下
- 关键改动内容（未来实现时）：
  - 统一提示语与语气系统

#### 任务 4.2：打扰策略设计
- 目标：
  - 统一“什么时候提醒”和“怎么提醒”
- 任务内容：
  - 定义分级提醒模型：
    - 轻提示
    - 正常提醒
    - 强提醒
  - 明确低干扰模式、专注模式、聊天中、鼠标交互中的行为差异
- 完成标志：
  - 所有提醒行为都可归类
- 关键改动内容（未来实现时）：
  - 建立统一提醒策略

#### 任务 4.3：关键用户路径整理
- 目标：
  - 明确最重要的 3~5 条助手使用路径
- 任务内容：
  - 至少覆盖：
    - 聊天 → 回复 → 记忆 / 意图
    - 提醒 → 气泡 → 用户响应
    - 专注开始 → 阶段变化 → 结束反馈
    - 待办新增 / 完成 → 情绪反馈
- 完成标志：
  - 后续体验优化围绕核心路径，而不是散点打补丁
- 关键改动内容（未来实现时）：
  - 以关键路径驱动体验优化

#### 任务 4.4：多窗口体验一致性方案
- 目标：
  - 让主窗口、子窗口、气泡窗口被感知为同一个助手
- 任务内容：
  - 定义一致性规则：
    - 视觉反馈
    - 打开/隐藏时机
    - 互斥与联动规则
    - 过渡节奏
- 完成标志：
  - 后续 UI / 交互优化有统一判断依据
- 关键改动内容（未来实现时）：
  - 统一多窗口体验规范

#### 任务 4.5：体验验收口径设计
- 目标：
  - 为后续每轮体验优化建立可复用的验收标准
- 任务内容：
  - 建立体验验收维度：
    - 是否自然
    - 是否打扰过多
    - 是否符合人设
    - 是否连续一致
- 完成标志：
  - 后续不再用“感觉差不多”验收体验
- 关键改动内容（未来实现时）：
  - 建立轻量体验 review 清单

## 7. Layer 5：工程与质量层

### 7.1 层级目标

让项目从“能编过 + 靠人工确认”升级为“关键行为有最小自动化保障，回归成本可控”。

### 7.2 当前现状

- 已有 `check:ts`、`check:rust`、`check`、`verify`
- 当前自动检查更多是编译检查，不是行为验证
- 多窗口、透明、穿透、托盘、定位联动仍重度依赖人工验收
- 每次改动后的回归成本偏高

### 7.3 子任务清单

#### 任务 5.1：质量防线分层设计
- 目标：
  - 明确哪些问题由编译检查兜底，哪些由测试兜底，哪些只能人工验收
- 任务内容：
  - 定义三层质量防线：
    - 编译/静态检查
    - 自动化行为检查
    - 人工桌面验收
- 完成标志：
  - 后续不再把所有验证都压给人工
- 关键改动内容（未来实现时）：
  - 按质量类型配置验证手段

#### 任务 5.2：最小自动化测试优先级设计
- 目标：
  - 决定第一批最值得自动化的逻辑
- 任务内容：
  - 优先级建议：
    1. service 层纯逻辑
    2. AI 后台任务决策
    3. 事件桥接逻辑
    4. Tauri command 边界
- 完成标志：
  - 后续自动化程序可直接按优先级拆测试任务
- 关键改动内容（未来实现时）：
  - 先补高价值低成本测试

#### 任务 5.3：人工回归重点收敛
- 目标：
  - 明确哪些场景必须人工看，哪些不应再重复人工验证
- 任务内容：
  - 保留人工重点：
    - 透明
    - 穿透
    - 托盘
    - 多窗口定位联动
    - 交互动画
- 完成标志：
  - 每次改动都能用固定人工回归集
- 关键改动内容（未来实现时）：
  - 减少低价值重复人工回归

#### 任务 5.4：交付纪律方案
- 目标：
  - 让每轮开发结束时工作区、验证结果、变更范围都可解释
- 任务内容：
  - 定义提交前固定动作：
    - `git status`
    - 最小检查
    - 变更范围确认
    - 文档同步确认
  - 定义自动化程序 finalizer 收口口径
- 完成标志：
  - 后续每轮改动都能稳定收口
- 关键改动内容（未来实现时）：
  - 强化交付纪律与 finalizer 规则

#### 任务 5.5：质量度量方案
- 目标：
  - 为后续质量改善建立可跟踪指标
- 任务内容：
  - 建议跟踪：
    - 回归时间
    - 人工回归项数量
    - 每轮意外脏文件数量
    - 测试覆盖的关键模块数量
- 完成标志：
  - 后续质量优化不再只是主观印象
- 关键改动内容（未来实现时）：
  - 建立最小质量仪表盘

## 8. Layer 6：商业与扩展层

### 8.1 层级目标

在不急于做插件化和商业化实现的前提下，先定义未来能力扩展边界，避免产品成长时再次回到单体堆叠。

### 8.2 当前现状

- 当前产品仍以单体助手形态为主
- `cloudpet-mcp/` 存在，但主项目与扩展能力的系统边界还不够产品化
- 若未来扩到角色系统、插件、付费能力、更多 AI 工具调用，当前结构预留不足

### 8.3 子任务清单

#### 任务 6.1：核心能力边界定义
- 目标：
  - 明确哪些能力永远属于“助手核心”
- 任务内容：
  - 建议核心能力包括：
    - 桌面存在
    - 对话
    - 提醒
    - 待办 / 专注
    - 记忆
- 完成标志：
  - 后续讨论扩展时不会误动产品核心
- 关键改动内容（未来实现时）：
  - 固化核心能力边界

#### 任务 6.2：扩展能力分类方案
- 目标：
  - 明确哪些能力未来可扩展、可外挂、可订阅
- 任务内容：
  - 分类未来扩展：
    - MCP / 工具能力
    - 第三方服务整合
    - 新角色 / 新人设
    - 高级提醒 / 专注增强
- 完成标志：
  - 后续不再把所有新能力直接塞进主项目
- 关键改动内容（未来实现时）：
  - 建立扩展能力分类体系

#### 任务 6.3：能力接口边界设计
- 目标：
  - 为未来插件化 / 可配置能力预留接口概念模型
- 任务内容：
  - 定义未来接口需描述：
    - 输入
    - 输出
    - 触发条件
    - UI 表现权限
    - 数据访问权限
- 完成标志：
  - 后续做扩展时有清晰起点
- 关键改动内容（未来实现时）：
  - 定义扩展接口模型

#### 任务 6.4：商业化切口评估方案
- 目标：
  - 识别未来最可能承载付费或分层价值的能力
- 任务内容：
  - 评估维度：
    - 高频刚需
    - 体验升级感
    - 成本可控
    - 是否破坏核心完整性
- 完成标志：
  - 后续商业设计不会反向伤害产品结构
- 关键改动内容（未来实现时）：
  - 区分免费核心与增强能力

#### 任务 6.5：长期演进路线设计
- 目标：
  - 为未来 2~3 个阶段的产品演进预留结构路线
- 任务内容：
  - 划分：
    - 单体成熟期
    - 可扩展能力期
    - 商业化探索期
  - 明确每阶段不该提前做的事
- 完成标志：
  - 后续不会因为想太远而过早复杂化当前项目
- 关键改动内容（未来实现时）：
  - 形成长期演进路线图

## 9. 跨层执行规则

### 自动化程序后续拆任务时必须遵守

- 一次只推进一个层面
- 一个层面内部优先按依赖顺序推进
- 未完成上一层的边界定义，不进入下一层实现
- 每个实现型 story 必须注明：
  - 属于哪一层
  - 对应哪些子任务
  - 明确不属于哪一层，防止顺手扩散

### 每层完成定义

每层不能以“讨论过了”作为完成标准，必须至少满足：

- 层级目标已锁定
- 子任务边界已锁定
- 完成标志已锁定
- 依赖关系已锁定

这样自动化程序才能继续细化为 story / 执行单元。

## 11. 备注与假设

- 当前最优先目标是建立长期可持续的助手产品结构，而不是追求短期功能堆叠
- 六层是治理视角，不是代码目录；后续自动化程序可以按这套层级映射到代码、文档、测试任务
- 短期最重要的是前 3 层：架构与代码、事件与状态、AI 交互
- 产品体验、工程质量、商业扩展三层应建立在底层结构更稳定之后
